# Isarから別ライブラリへの移行計画

## 1. プロジェクト内での Isar の使用状況の調査結果

本プロジェクト（MeRead）では、RSSフィードや投稿データのローカル保存に `isar` および `isar_flutter_libs`（バージョン `^3.1.0+1`）を使用しています。主な使用状況は以下の通りです。

### 1.1. データモデルとリレーション（1対多、多対1の関係）
以下の3つの主要なコレクションが存在し、`IsarLink` および `IsarLinks`（`@Backlink`含む）を使用して関係が構築されています。
* **`Feed`** (フィード情報)
  * `Category` との関係: `IsarLink<Category>` (1つのフィードが1つのカテゴリに属する)
  * `Post` との関係: `IsarLinks<Post>` (バックリンク: 1つのフィードが複数の投稿を持つ)
* **`Post`** (記事・投稿情報)
  * `Feed` との関係: `IsarLink<Feed>` (1つの投稿が特定のフィードに属する)
* **`Category`** (カテゴリ情報)
  * `Feed` との関係: `IsarLinks<Feed>` (バックリンク: 1つのカテゴリが複数のフィードを持つ)

### 1.2. データベース操作（CRUDとクエリ）
`lib/helpers/isar_helper.dart` に主な操作が集約されています。
* **同期的なトランザクションの利用**: `_isar.writeTxnSync()` を用いて、データの保存(`putSync`, `putAllSync`)や削除(`deleteSync`, `deleteAllSync`)を同期的に行っています。
* **フィルタリングと検索**:
  * 完全一致検索: `urlEqualTo`, `nameEqualTo` などを利用。
  * リレーション越しの検索: `.feed((f) => f.idEqualTo(feed.id))` のように、リンク先のデータ条件に基づくフィルタリングを実行。
  * 全文検索ライクな検索: `titleContains(value).or().contentContains(value)` による部分一致検索を利用して記事の検索を実装しています。
* **ソート**: `.sortByPubDateDesc()` を用いて日付順の取得を行っています。

## 2. 移行先ライブラリの検討（2026年も見据えた選定）

Isarは非常に高速で優れたNoSQLデータベースでしたが、開発者がプロジェクトから離れ（現在はコミュニティフォーク版が存在する状態）、将来的なメンテナンス性に懸念があります。2026年でも安心して使用できる（活発にメンテナンスされている）代替ライブラリとして、以下の3つを検討しました。

### 候補1: Drift (旧 Moor)
SQLiteをベースにした、Dart/Flutter向けの強力なリレーショナルデータベース（ORM）です。
* **メリット**:
  * Simon Binder氏を中心に非常に活発にメンテナンスされており、将来性（2026年以降）も極めて高い。
  * SQLiteベースのため、複雑なクエリやデータ構造の変更に強く、公式ドキュメントも非常に充実。
  * Webを含めた全プラットフォームをサポート。
* **デメリット**:
  * Isar（NoSQLベース）とはパラダイムが異なり、SQLリレーショナルな設計への見直しが必要（テーブル設計、外部キー制約、JOINによるデータ取得などへの書き換え）。
  * Isarの `IsarLink` のような直感的なオブジェクトのネストではなく、JOINを書く必要があるため、移行時のコード変更量がやや多くなる。

### 候補2: ObjectBox
C言語/C++で書かれた超高速なNoSQLのエッジデータベースで、ObjectBox社によって商用レベルでメンテナンスされています。
* **メリット**:
  * 非常に高速で、Isarと同じくNoSQL・オブジェクトベースのアプローチを取るため、Isarからの概念的な移行が最もスムーズ。
  * リレーション機能（`ToOne`, `ToMany`）が充実しており、現在の `IsarLink`, `IsarLinks` の構造をほぼそのままマッピングできる。
  * 開発元が企業であり、継続的なアップデートとサポートが期待できる。
* **デメリット**:
  * オープンソース（OSS）ではなくプロプライエタリなコアを持つ（一部OSS化の動きはあり）。
  * Webのサポートに制限がある（主にモバイル・デスクトップ向け。ただし本アプリはモバイル向けが主軸のため許容範囲と考えられる）。

### 候補3: Realm (MongoDB)
MongoDBが提供するモバイル向けデータベースです。
* **メリット**:
  * リレーションやオブジェクトベースの操作が可能。
* **デメリット**:
  * Atlas Device Syncが2024年に非推奨（Deprecation）となり、コミュニティ版への移行が推奨されている状況。将来性やエコシステムの安定性に不安が残るため、今回のユースケースには適さない。

## 3. 結論と推奨ライブラリ

本プロジェクトへの移行先としては、以下のいずれかを推奨します。

**【第一推奨】ObjectBox**
* **理由**: Isarと同じNoSQL型であり、`IsarLink` を `ToOne` / `ToMany` に置き換えるだけでデータモデルの移行がしやすく、書き換えコストが低いためです。また、パフォーマンスもIsarと同等以上に高く、モバイルアプリのローカルデータベースとして非常に強力です。商用企業がバックアップしているためメンテナンス性も担保されます。

**【第二推奨】Drift**
* **理由**: もし将来的にWeb対応を強力に推進したい場合や、完全にオープンソースのエコシステムに依存したい場合はDriftがベストです。SQL（リレーショナル）へのパラダイムシフトが必要ですが、最も安定しており「2026年でも絶対にメンテナンスされている」という安心感は随一です。

現在の `IsarHelper.dart` の実装を見る限り、複雑なSQLを必要とする処理は少なく、オブジェクトの保存とシンプルなリレーションによるクエリが主であるため、実装の手軽さを重視するのであれば **ObjectBox** への移行が最も効率的です。

## 4. 移行に向けたステップ (ObjectBoxを選択した場合の例)

1. **パッケージの置き換え**:
   * `pubspec.yaml` から `isar`, `isar_flutter_libs`, `isar_generator` を削除。
   * 代わりに `objectbox`, `objectbox_flutter_libs` と `build_runner` 用のパッケージを追加。
2. **モデルの修正**:
   * `@collection` アノテーションを `@Entity()` に変更。
   * `Id? id = Isar.autoIncrement;` を `@Id() int id = 0;` のようなObjectBoxのID仕様に変更。
   * `IsarLink<T>` や `IsarLinks<T>` を `ToOne<T>` と `ToMany<T>` に変更。
3. **ヘルパークラスの修正 (`isar_helper.dart` -> `db_helper.dart`)**:
   * `Isar.open` を ObjectBox の `Store` の初期化処理に置き換え。
   * `_isar.writeTxnSync()` などを ObjectBox の `Box` クラスを利用した `box.put()`, `box.remove()` 等の操作に書き換え。
   * クエリ操作（`filter()`, `titleContains` など）を ObjectBox のクエリビルダー (`box.query(Post_.title.contains(value)).build()`) に書き換え。
4. **既存データへの対応（必要であれば）**:
   * 既存ユーザーのIsarデータを新データベースへ移行するマイグレーションロジックの作成（Isarと新DBを両方読み込み、移し替えた後にIsarのファイルを削除する処理）。不要な場合は新規インストール同様に扱う。
